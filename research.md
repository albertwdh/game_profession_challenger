## Gemini 说

开发一款2D职场模拟游戏是一个极具传播潜力的方向。这类游戏的核心魅力不在于画面有多华丽，而在于**强烈的代入感、黑色幽默以及对现实的解构**。

为了确保你作为独立开发者能够高效落地，我们需要把目标收敛到一个**最小可行性产品（MVP）**。MVP 的核心目标是：验证核心玩法（Core Loop）和文案是否能引发玩家共鸣并产生分享欲。

以下是为你量身定制的 2D 职场模拟游戏 MVP 方案与执行路径：

### 一、 MVP 核心玩法与数值架构设计

放弃复杂的动作操作，采用**“资源管理 + 随机事件抉择”**的轻量级玩法（参考《王权 Reigns》或《中国式家长》的文字选项模式）。

**1. 核心四大核心资源（决定生死）：**

- 💰 **存款：** 归零则破产，游戏结束（Bad Ending: 流落街头）。
- ❤️ **精神状态（SAN值/头发）：** 归零则抑郁或猝死，游戏结束（Bad Ending: 住进ICU或进山出家）。
- 💼 **业务能力：** 影响升职加薪的概率。
- 🐟 **摸鱼值：** 过高会被开除，适度摸鱼可以恢复精神状态。

**2. 核心游戏循环（Core Loop）：** 每天（回合）开始 -> 触发日常工作任务 -> 触发随机/突发事件 -> 玩家做出选择 -> 结算四大资源增减 -> 下班（可选择加班、娱乐、学习）-> 进入下一天。

------

### 二、 职业设定与事件池库（剧本核心）

MVP 阶段不要做太多职业，先做 **3个极具话题性** 的职业打透。每个职业的事件库分为四个维度：日常（乏味）、高光（有趣）、社死（尴尬）、离谱（搞笑）。

#### 职业 A：全栈程序员 / 研发工程师

- **日常（乏味）：** 参加没有意义的对齐会；盯着进度条看依赖包下载；排查一个因为拼写错误导致的 Bug 找了一下午。
- **有趣（高光）：** 写了一段极其优雅的代码，一次编译通过（SSR级罕见事件，精神状态大涨）；利用自动化脚本把一天的工作一小时干完，剩下的时间光明正大摸鱼。
- **尴尬（社死）：** 在亲戚聚会上被要求帮忙修电脑或黑进别人的微信；演示 MVP 产品时，平时好好的系统突然当众崩溃报错。
- **搞笑（离谱）：** 实习生不小心执行了 `DROP TABLE` 删库跑路；深夜被报警电话叫醒，发现是机房网线被老鼠咬了；虔诚地给服务器贴上“逢考必过”的符咒。

#### 职业 B：新媒体运营 / 社交平台博主

- **日常（乏味）：** 每天盯着数据大盘，为了几个点击量掉头发；被老板要求用 0 预算做出 10 万+ 的爆款。
- **有趣（高光）：** 偶然发了一条吐槽公司的动态，结果在小红书/抖音上意外爆火，涨粉十万；接到第一个暗广，赚到外快。
- **尴尬（社死）：** 忘记切换账号，用公司官方号在美女/帅哥视频下发了“贴贴”；精心策划的图文被老板批得一无是处，随便乱发的却数据极好。
- **搞笑（离谱）：** 发现市面上居然有自动生成文案、配图并自动唤醒 App 发帖的 AI 神器，于是你开始每天只工作 5 分钟，剩下的时间提心吊胆怕被老板发现。

#### 职业 C：刚入职的管培生 / 职场萌新

- **日常（乏味）：** 帮全组人贴发票报销；点下午茶奶茶；整理根本没人看的会议纪要。
- **有趣（高光）：** 凭借初生牛犊不怕虎的气势，在会议上直接“整顿职场”，硬刚画大饼的领导（可能导致高额精神奖励，但也可能直接被开除）。
- **尴尬（社死）：** 在公司大群里发吐槽老板的截图，结果发错了群；电梯里遇到大老板，紧张到按错了楼层。
- **搞笑（离谱）：** 发现全公司只有自己一个人在认真干活，其他人全在熟练地切屏摸鱼；为了融入前辈的圈子，硬着头皮学会了看懂复杂的股市 K 线图。

### 三、 技术栈建议与系统架构

基于高效开发和快速部署的原则，建议采用以下技术栈组合：

**1. 前端（游戏表现层）：**

- 强烈建议使用 **React** 结合简单的 CSS 动画，或者引入轻量级的 2D 游戏库（如 `Phaser.js`）。对于这种重 UI 和事件流的游戏，React 的状态管理机制处理起来得心应手。
- 如果是移动端优先，也可以考虑直接打包成 PWA，或者用跨平台框架包一个壳。

**2. 后端与接口（数据处理层）：**

- **单机为主，弱联网：** MVP 阶段可以不需要复杂的后端。但如果你想加入“全服玩家平均存活天数”、“奇葩死法排行榜”等增加社交传播度的功能，可以用 **Go (Gin框架)** 写几个极简的 RESTful API。Go 的高并发处理排行榜非常合适。
- 数据存储使用简单的 MySQL 或 SQLite 即可。

**3. 数据配置管理：**

- 游戏内的海量文案和事件，**绝对不要硬编码在代码里**。
- 可以使用 Python 配合 Pandas 编写一个处理脚本，将 Excel 中策划好的事件表格（包含事件描述、选项 A/B、资源增减数值）直接转换为 JSON 格式，前端在初始化时加载这个 JSON 字典。这会极大地提升你后续更新内容的效率。

### 一、 整体技术架构设计

我们将系统拆分为三个核心模块：**前端表现层**、**数据驱动管线**（核心）和**后端服务层**。

#### 1. 数据驱动管线（Content Pipeline）

这是整个游戏的心脏。游戏里的几百个随机事件绝对不能写死在代码里，否则后续维护会让人崩溃。

- **配置中心：** 使用 Excel 统筹管理所有的职业、事件、选项和数值增减。
- **自动化转换：** 编写一个 **Python + Pandas** 脚本，读取策划好的 Excel 表格，清洗数据后，自动生成前端需要的 `events.json`。
- **优势：** 你可以随时修改文案、调整“精神状态”的掉血比例，只需跑一下脚本生成新的 JSON，前端零代码修改即可生效。

#### 2. 前端表现层 (SPA Web App)

由于是 2D 文字和数值驱动游戏，复杂的物理引擎是多余的。

- **核心框架：** 使用 **React**。利用 React 强大的状态管理来驱动游戏循环。
- **状态管理：** 核心维护一个 `GameState`（包含四大资源数值：存款、精神、能力、摸鱼值，以及当前天数、当前职业）。
- **UI 风格：** 建议采用极简的“复古像素风”或“Windows 95 弹窗风”，配合简单的 CSS 动画（比如扣除精神值时的屏幕震动），开发成本极低且自带梗属性。

#### 3. 后端服务层 (RESTful API)

MVP 阶段的后端主要负责用户鉴权、存档管理和排行榜（制造社交传播锚点）。

- **核心框架：** 使用 **Go (Gin框架)**，轻量、高并发，非常适合做排行榜接口。
- **数据库：** MySQL 存储用户基础信息和最终游戏结算数据。
- **鉴权：** **JWT (JSON Web Token)**，用户无需复杂注册，甚至可以一键匿名登录生成 Token。
- **部署：** 编译打包后部署到 Linux 服务器，前端静态资源和后端接口统一由 **Nginx** 进行反向代理和路由分发。

### 二、 MVP 执行路径 (两周冲刺计划)

我们将开发划分为四个清晰的里程碑。

#### 里程碑 1：定义数据标准与处理脚本 (Day 1-2)

**目标：跑通从 Excel 到 JSON 的数据管线。**

1. **制定 Excel 表头：**
   - `EventID` (事件唯一ID)
   - `Occupation` (所属职业，如：全栈开发)
   - `Description` (事件文案：比如“今天产品经理要求加一个五彩斑斓的黑”)
   - `OptionA_Text` (选项A文案：硬刚)
   - `OptionA_Effect` (选项A数值影响：存款+0, 精神+20, 能力+0, 摸鱼-10)
   - `OptionB_Text` (选项B文案：默默加班实现)
   - `OptionB_Effect` (选项B数值影响：存款+50, 精神-30, 能力+10, 摸鱼+0)
2. **编写 Python 脚本：** 使用 Pandas 读取表格，处理空值，导出为结构化的 JSON 文件。

#### 里程碑 2：前端核心引擎与闭环 (Day 3-6)

**目标：能在本地浏览器里把游戏完整“玩”一遍。**

1. **初始化 React 项目：** 搭建基础目录，引入上一步生成的 `events.json`。
2. **构建全局状态机：** 初始化四大资源（存款 1000，精神 100，能力 10，摸鱼 0）。
3. **编写 Core Loop 组件：**
   - 开发 `EventCard` 组件：展示事件描述。
   - 开发 `OptionButton` 组件：点击后触发数值结算逻辑。
   - 编写每日结算逻辑：如果某项数值归零（如精神<=0），跳转到 `GameOver` 页面（猝死结局）。
4. **跑通单机闭环：** 玩家可以一直点选项，直到触发结局，实现真正的 MVP。

#### 里程碑 3：后端 API 与联调 (Day 7-10)

**目标：加入账号系统和排行榜，为社交裂变做准备。**

1. **搭建 Gin 服务：** 初始化 Go Module，配置路由组。
2. **实现鉴权：** 编写 JWT 生成和解析中间件。提供 `/api/login`（匿名注册/登录）接口。
3. **结算与排行榜：**
   - 开发 `/api/game_over` 接口：接收前端传来的存活天数和死法。
   - 开发 `/api/leaderboard` 接口：返回全服“最长寿打工人”或“最惨死法”排行榜。
4. **前后端联调：** 前端在游戏结束后，将战绩 POST 到后端，并拉取排行榜展示。

#### 里程碑 4：部署上线与打磨 (Day 11-14)

**目标：发布公网可访问的版本。**

1. **环境配置：** 在 Linux 服务器上配置 MySQL 和 Nginx。
2. **打包部署：** React 执行 Build 生成静态文件丢进 Nginx 目录；Go 编译为二进制文件在后台通过 systemd 或 nohup 运行。
3. **Nginx 路由：** 配置 `/` 指向 React 静态文件，`/api` 反向代理到 Go 服务的端口。
4. **内测找 Bug：** 发给几个朋友试玩，重点测试数值是否容易崩盘（比如一直选某个选项就能长生不老）。
